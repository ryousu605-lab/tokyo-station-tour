## 問題 1: 回転ボタンが無限に回り続け、向いている方角が分かりづらい

### 現状の動作と原因

- 画面端の「‹ ›」ボタンは `turnView(step)`（virtual-tour.html 内）で実装されており、
  **写真のある方角だけを円環（ループ）として巡回**する。
  ```js
  // 現状: 端が無く、押し続けると永遠に一周し続ける
  setDir(dirs[(i + step + dirs.length) % dirs.length]);
  ```
- 写真の無い方角は**スキップ**されるため、例えば東西 2 枚しか写真が無い地点
  （`b1_yaesu`, `north_center`, `maru_central` など）では「›」を押すたびに
  東⇄西が 180° 反転し続ける。演出（14% の横スライド）は 90° 回転時と同一なので、
  ユーザーは何度回ったのか・いまどちらを向いているのかを見失う。
- コンパスも写真の無い方角のボタンを `display:none` で**完全に消している**ため、
  「全体のうちどの方角が欠けているか」という手がかりも無い。

### 修正方針: 円環をやめ「弧（アーク）」で端に突き当たったら止める

写真のある方角を時計回りに並べ、**写真の無い区間のうち最大の空きで切り開いた弧**を
回転範囲とする。弧の端に達したらそれ以上回転せず、そちら側のボタンを無効化する。
全 4 方角の写真がある地点だけは従来どおり一周できる（制限の必要が無いため）。

例:
| 地点 | 写真のある方角 | 回転範囲（弧） | 端での挙動 |
|---|---|---|---|
| `central_yaesu` | 北東南西（全部） | 制限なし（一周可） | — |
| `b1_gransta` | 北・南・西 | 南→西→北 | 北で「›」無効、南で「‹」無効 |
| `north_yaesu` | 東・南 | 東→南 | 90°の範囲だけ往復 |
| `b1_yaesu` | 東・西 | 東→西（一方向のみ1回） | 反転は1回きり。連打で回り続けない |
| `chika_escalator` | 東のみ | 回転不可 | ボタン非表示（従来どおり） |

### 実装詳細（virtual-tour.html）

1. **弧の算出と回転先の決定**（`turnView` の置き換え、`availableDirs` の直後あたりに追加）

   ```js
   // 写真のある方角を「弧」として返す。全4方角あれば null（=一周可）。
   function turnArc(id){
     const dirs = availableDirs(id);          // DIRS 順 = 時計回り
     if(dirs.length >= 4 || dirs.length <= 1) return null;
     let cut = 0, gapMax = -1;                // 時計回りで最大の「写真の無い区間」を探す
     for(let i = 0; i < dirs.length; i++){
       const a = COMPASS_FROM_N[dirs[i]];
       const b = COMPASS_FROM_N[dirs[(i + 1) % dirs.length]];
       const gap = (b - a + 360) % 360;
       if(gap > gapMax){ gapMax = gap; cut = (i + 1) % dirs.length; }
     }
     return dirs.slice(cut).concat(dirs.slice(0, cut));   // 空きの直後から並べた弧
   }

   // step(-1/+1) 方向の回転先。弧の端なら null（=回転不可）。
   function turnTarget(step){
     const dirs = availableDirs(current);
     if(dirs.length <= 1) return null;
     const arc = turnArc(current);
     if(!arc){                                 // 全方角あり: 従来どおり一周
       const i = dirs.indexOf(viewDir);
       return dirs[(i + step + dirs.length) % dirs.length];
     }
     const i = arc.indexOf(viewDir) + step;
     return (i < 0 || i >= arc.length) ? null : arc[i];
   }

   function turnView(step){
     const d = turnTarget(step);
     if(!d) return;
     turnStep = step * Math.max(1, Math.round(dirGap(viewDir, d) / 90)); // 180°反転はスライド2倍
     setDir(d);
   }
   ```

2. **ボタンの無効表示**（`renderCompass` 内の turnBtn 表示制御を差し替え）

   ```js
   document.querySelectorAll(".turnBtn").forEach(b => {
     b.style.display = dirs.length > 1 ? "flex" : "none";
     b.disabled = turnTarget(b.classList.contains("left") ? -1 : 1) === null;
   });
   ```
   `setDir()` は既に `renderCompass()` を呼ぶため、回転のたびに無効状態が更新される。

   CSS 追加:
   ```css
   .turnBtn:disabled{opacity:.25; cursor:default;}
   .turnBtn:disabled:hover{background:rgba(28,27,25,0.45);}
   ```

3. **コンパスの改善（方向感覚の補助）**
   写真の無い方角のボタンを `display:none` で消すのをやめ、**薄く表示**する
   （`opacity:.18` 程度・従来どおり押せない）。4 文字が常に同じ位置に見えることで
   「どの方角が欠けているか」「いまどこを向いているか」が一目で分かる。

   ```js
   b.style.display = "flex";
   b.style.opacity = dirs.includes(b.dataset.dir) ? "" : ".18";
   ```

4. **スライド演出の補正**（`swapLayers`）
   `turnStep` を上記 1. で「またいだ 90° の数」倍にしたため、180° 反転時は
   28% スライドになり、90° 回転との違いが視覚的に分かる。`swapLayers` 側の変更は不要
   （`off = turnStep * 14` のまま）。

5. **使い方テキストの更新**（`#howtoPanel`）
   「未撮影の方向は回転で飛ばします」→「写真の無い方向へは回転できません（端で止まります）」。

### 受け入れ条件（問題 1）

- [ ] 全 4 方角の写真がある地点（`yaesu_central` 等）では従来どおり一周できる
- [ ] 写真が欠けた地点では弧の端で回転が止まり、その側のボタンが薄い無効表示になる
- [ ] 無効ボタンを押しても写真・コンパス・矢印が一切変化しない
- [ ] 写真 1 方角のみの地点では回転ボタンが非表示（従来どおり）
- [ ] 180° 反転時のスライド量が 90° 回転の 2 倍になっている
- [ ] コンパスに 4 方角が常時表示され、写真の無い方角は薄く見える
- [ ] 地点移動（`goTo`/`goBack`）後の自動方向合わせ（`faceTravelDir`→`snapDir`）が従来どおり動く

## 問題 2: 地点間の接続が順番どおりでなく「飛び飛び」になっている

### 監査結果（マップ座標に基づく機械検査）

全 27 地点・全リンクについて「リンク線が別のピンのすぐ脇（距離 3% 未満）を素通りしていないか」
「最近傍の地点と未接続でないか」を検査した結果、以下の 3 箇所を検出:

| # | 現在のリンク | 素通りしている中間地点 | 距離 |
|---|---|---|---|
| A | `north_yaesu` (31, 42) ↔ `shinkansen_north` (33.7, 55.6) | `north_center` (31, 56) ※最近傍 d=2.7 | — |
| B | `yaesu_south_shinkansen_bento` (72.2, 54.9) ↔ `yaesu_south_bento` (78.9, 37.8) | `yaesu_south_shinkansen` (75, 45.9) | 0.68 |
| C | `chika_escalator` (51.5, 69.4) ↔ `maru_central` (51.3, 79.6) | `central_maru` (52.3, 75.7) | 0.92 |

片方向のみのリンク（戻れない接続）は検出されなかった。

### 修正 A: 新幹線北乗換口は「北通路 中程」に接続する

ピン位置上、`shinkansen_north` は `north_center`（北通路 中程）の真横にある。
`north_yaesu`（北通路 八重洲寄り）とのリンクは通路を 14% 分も飛ばしていた。

`scenes.js` の変更:

- `north_yaesu.links` から `{to:"shinkansen_north", dir:"left", ...}` を**削除**
- `north_center.links` に**追加**:
  ```js
  {to: "shinkansen_north", dir: "left", label: "新幹線北乗換口へ", x: 15, y: 58}
  ```
- `shinkansen_north.links` の戻り先を変更:
  ```js
  {to: "north_center", dir: "back", label: "北通路へ戻る", x: 50, y: 86}
  ```

### 修正 B: 八重洲南口チェーンを地図の並び順どおりに直列化する

現状は `…新幹線口弁当屋 → 弁当屋 → 新幹線（南寄り・行き止まり）` の順だが、
地図上の並びは `新幹線口弁当屋 → 新幹線（南寄り） → 弁当屋` であり、
現リンクは中間の「新幹線（南寄り）」のほぼ真上を素通りしている。

`scenes.js` の変更（チェーンを `yaesu_south_shinkansen_bento ↔ yaesu_south_shinkansen ↔ yaesu_south_bento` に）:

- `yaesu_south_shinkansen_bento.links` の forward 先を変更:
  ```js
  {to: "yaesu_south_shinkansen", dir: "forward", label: "八重洲南口周辺（新幹線・南寄り）へ", x: 50, y: 45}
  ```
- `yaesu_south_shinkansen.links` を差し替え（行き止まり解消・中継点化）:
  ```js
  links: [
    {to: "yaesu_south_bento", dir: "forward", label: "八重洲南口周辺 弁当屋へ", x: 50, y: 45},
    {to: "yaesu_south_shinkansen_bento", dir: "back", label: "新幹線口弁当屋へ戻る", x: 50, y: 86}
  ]
  ```
- `yaesu_south_bento.links` を差し替え（終端化）:
  ```js
  links: [
    {to: "yaesu_south_shinkansen", dir: "back", label: "新幹線（南寄り）へ戻る", x: 50, y: 86}
  ]
  ```

### 修正 C: 地下入口エスカレーターの接続（要判断・推奨案あり）

`chika_escalator` のピン (51.5, 69.4) は中央通路の `central_center` (52.3, 63.1) と

**推奨（マップ準拠で張り替え）** — ユーザー指示「マップ上の地点を参考にして正確に接続」に従う:

- `maru_central.links` から `{to:"chika_escalator", dir:"forward", ...}` を**削除**
- `central_maru.links` に**追加**:

- `chika_escalator.links` の戻り先を変更:
  ```js
  {to: "central_maru", dir: "back", label: "中央通路（丸の内寄り）へ戻る", x: 50, y: 86}
  ```

- `b1_ginsuzu` の up リンクのラベルを実態に合わせて微修正:
  「1F 丸の内中央口方面へ（エスカレーター）」→「1F 中央通路（丸の内寄り）へ（エスカレーター）」

**代替案**: 実際のエスカレーターが丸の内中央口のすぐ横にあるなら、リンクは現状のまま

※どちらか一方のみ実施。撮影時の記憶と現地の位置関係で判断する。

## 影響範囲・リスク

- **データ変更は 7 地点の links のみ**。座標・写真・PHOTOS マニフェストは変更しない
  （代替案 C' を選んだ場合のみ `chika_escalator` の map 座標が変わる）
- 矢印の向き・位置・下部チップはすべてマップ座標から自動計算されるため、
  リンク張り替えに伴う手動の座標調整は不要（階段リンクの x,y のみ据え置き）
- `history`/`visited`（戻るボタン・軌跡）はシーン ID ベースなので影響なし
- 回転ロジックの変更は `turnView` 経由のみ。`goTo` 到着時の `snapDir` は従来コードを共有し続ける
