## Phase A — 地点の変更・追加（要件①）

### A-1. データの一元化（K1・K2 の解消、最初に実施）

1. `virtual-tour.html` から `const MAPS = {...}` / `const startId = ...` / `const scenes = {...}` の `<script>` ブロックを削除し、メイン `<script>` の直前に `<script src="scenes.js"></script>` を追加する。
2. `scenes.js` を**唯一の正**とする。HTML 内のコメント（544〜547行目の「単一ファイル化」の説明）と howtoPanel の「位置の微調整」記述を scenes.js 前提に書き換える。
3. ローカル確認は file:// 直開きではなく、プロジェクトフォルダで `python -m http.server 8000` → `http://localhost:8000/virtual-tour.html` で行う（README 相当として撮影リスト.md に1行追記）。

これにより編集フローが「`?edit` で調整 → scenes.js 書き出し → ファイル置換 → `git push`（Vercel 自動デプロイ）」で完結する。

> 注意: `exportScenes()` は JSON.stringify 出力のためコメント（`/* ---- 八重洲側 ---- */` 等）が消える。コメントを保持したい場合は書き出しを「差分の手動反映」に使い、scenes.js 本体はエディタで編集する運用とする（どちらでも動作は同一）。

### A-2. 既存地点の「変更」手順（定型ワークフロー）

| 変更内容 | 手順 |
|---|---|
| 名称・説明文 | `scenes.js` の `name` / `desc` を直接編集 |
| 構内図上のピン位置 | `?edit` で全体マップを開き、正しい位置をタップ → `map(1F): {x: NN.N, y: NN.N}` がコピーされる → `map:` に貼り付け |
| 矢印（リンク）の写真上の位置 | 該当地点を表示して `?edit` でホットスポットをドラッグ → 書き出しに反映される（up/down リンクのみ有効。平面リンクは座標から自動配置のため調整不要） |
| リンクの追加・削除 | `links` 配列を編集。**必ず双方向**（A→B を足したら B→A も足す）。`dir` は平面移動なら任意（`forward` 等でよい。矢印は自動計算）、階移動のみ `up`/`down` を正しく指定 |
| 地点の削除 | エントリ削除に加え、**他地点の links から当該 ID への参照を全削除**（残すと Phase C-1 の検証でエラーになる） |

### A-3. 地点の「追加」手順（雛形）

```js
新地点ID: {
  name: "表示名",
  desc: "説明文。",
  floor: "1F",              // "1F" | "B1" 必須
  map: {x: 0, y: 0},        // ?edit の地図タップで取得した値
  dest: false,              // 目的地（4方向撮影する場所）なら true
  links: [
    {to: "隣接地点ID", dir: "forward", label: "◯◯へ", x: 50, y: 45}
  ]
},
```

チェックリスト（追加のたびに全項目確認）:
1. ID は半角英数＋アンダースコア（写真ファイル名になるため）
2. `floor` 必須（ミニマップ・全体マップの切替に使用）
3. 隣接地点側にも逆向きリンクを追加
4. `撮影リスト.md` に行を追加し、合計枚数を更新（dest: true なら4枚、false なら1枚）
5. Phase C-1 の検証スクリプトを実行してエラーゼロを確認

### A-4. 追加候補（ユーザー決定事項）

以下は現行データの空白地帯。どれを採用するかを決めてから A-3 を実施する。

- 1F: 京葉線連絡通路方面、日本橋口方面、動輪の広場（B1丸の内地下南口の先）
- B1: 東京駅一番街（八重洲地下中央口の改札外）、北地下自由通路、京葉ストリート
- 改札外エリアを入れる場合は「改札内/外」の区別を `desc` に明記する（改札を通る移動である旨をリンクの `label` に含める）

### A-5. 地図マーキングによる一括追加（採用方式）

ユーザーが構内図の**コピー画像にマークを描くだけ**で、座標の取得と雛形エントリの生成を自動化する。`?edit` での1点ずつのタップ取得より、複数地点の追加に向く。

手順:
1. `maps/1f.png`（または `b1.png`）をコピーして `marks_1f.png` を作り、ペイント等で追加したい位置に**塗りつぶしの丸マーカー**を描く。
   - 色の規約: **純赤 `#FF0000`** = 目的地（dest: true）／**純青 `#0000FF`** = 通過点（dest: false）。構内図に存在しない純色を使うことで機械検出を確実にする
   - 丸の直径は 30px 以上。マーカー同士は重ねない
   - 各マーカーの脇に番号を書き（検出には使わないので手書きでよい）、別途「番号 → 地点名・説明」の対応メモを用意する
2. マーク済み画像を Claude に渡す → 検出スクリプト（Python/Pillow: 指定色ピクセルの連結成分の重心を算出）が各マーカーの `{x, y}` を%座標で確定する
3. Claude が雛形エントリ（A-3 の形式）を生成し、**最寄りの既存地点との相互リンク案**（構内図上の距離から算出）を添えて提示する
4. ユーザーが名称・リンク案を確認 → `scenes.js` に反映 → `validate.py` でエラー0件を確認 → push

既存地点の**位置修正**にも同じ方式が使える（例: 純緑 `#00FF00` = 「この既存ピンをここへ動かす」＋番号で既存 ID を指定）。

> なお `?edit` 内に地点追加 UI を作る案（v2 で非スコープとした GUI エディタ）は、本方式で代替できるため実装しない。

## Phase B — 撮影済み写真の一括割当（要件②）

**前提の変更**: 写真は**すでに撮影済みで、リスト順にはなっていない**。したがって「撮影順どおりに撮る」規約は使えない。並び替え作業を発生させずに割り当てる方式にする。

### B-1. 方式: ファイル名からの推定（採用）

**写真のファイル名に含まれる地点・方角の手がかりを辞書照合で解決する**方式を採用する。EXIF（撮影時刻・方位）は使わず、名前だけで機械的に割り当てる。

仕組み:
1. `scenes.js` の全地点から**照合辞書**を自動生成する。各地点 ID に対し、正式名称（`name`）・名称の部分文字列・想定される略称・ID そのものを対応付ける
   - 例: `yaesu_central` ← 「八重洲中央口 改札前」「八重洲中央口」「八重洲中央」「yaesu_central」
2. 方角は次の表記ゆれをすべて受け付ける: `北/N/n/kita` → `_n`、`東/E/e/higashi` → `_e`、`南/S/s/minami` → `_s`、`西/W/w/nishi` → `_w`
3. 各写真のファイル名（拡張子・連番・括弧・スペースを除去して正規化）を辞書と突き合わせ、`{地点ID}_{方角}.jpg` を決定する
   - マッチ例: `八重洲中央口 北.jpg` → `yaesu_central_n.jpg`／`ginsuzu_e (1).jpg` → `b1_ginsuzu_e.jpg`
4. 通過点（dest: false）は方角なしの名前（例 `南通路中程.jpg` → `south_center.jpg`）を許容する

判定結果は3分類で扱う: **確定**（辞書に一意マッチ）／**候補あり**（複数候補 or 曖昧一致 → 対応表を提示して選択してもらう）／**不明**（手がかりなし → ファイル名を提示して地点・方角を指定してもらう）。

> ファイル名にまだ地点の手がかりが無い写真が混ざっている場合は、先にエクスプローラー上で「地点名＋方角」が分かる名前に変更してもらう（形式は自由。上記の表記ゆれ吸収で拾える）。全ファイルを厳密な `{id}_{dir}.jpg` に直す必要はない — それはスクリプト側の仕事。

### B-2. 運用①（推奨・今回の一括分): Claude に割当を依頼する

ツールを作り込まず、この場で完了させる最短ルート。

1. ユーザー: 撮影済み写真をプロジェクト内の `incoming/` フォルダにまとめて入れる（いまのファイル名のままでよい）
2. Claude: 照合スクリプトを実行し、「元ファイル名 → 割当先（地点・方角）→ 判定（確定/候補/不明）」の**全件対応表**を提示する。「候補あり」「不明」の写真は中身も直接見て（看板・改札名から）推定候補を添える
3. ユーザー: 対応表を確認し、候補/不明分だけ回答する（確定分が正しければ「OK」だけでよい）
4. Claude: `photos/{id}_{dir}.jpg` へ**リネームコピー**（`incoming/` の元ファイルは残す）し、割当レポート `photos/assign_report.txt` を出力する
5. ローカル確認（`python -m http.server`）→ 4方向表示を目視で最終確認（北を向いたとき北の写真が出るか）→ push

同じ割当先が2枚以上ある場合（撮り直しの重複）は、両方を提示してどちらを採用するか確認する。割当先が1枚も無い地点は「欠落一覧」として報告する（C-1 の V8 と同じ観点）。

### B-3. 運用②（任意・次回以降の追加撮影用）: 割当ツール `tools/photo-sorter.html`

今後も写真を追加する場合に備えた自己完結ツール。ブラウザだけで動く。

1. ローカルで開き、写真を複数選択（またはドラッグ&ドロップ）
2. B-1 と同じ辞書照合をJSで行い、「確定/候補/不明」に分けてサムネイル付きで一覧表示
3. 候補・不明分は「地点」プルダウン（scenes.js からフロア別に生成）と N/E/S/W ボタンで指定・修正
4. 出力: File System Access API（Edge/Chrome）で `photos/` フォルダへ直接リネームコピー。非対応ブラウザ向けには `rename.ps1`（リネームコピーの PowerShell スクリプト）をダウンロード

運用①で今回分を済ませたあと、必要になった時点で実装する（今回のスコープ必須ではない）。

### B-4. 部分公開のサポート

割当が済んだ地点から順次 push してよい（未配置地点は現行どおりプレースホルダ表示）。全72枚が揃うまで公開を待つ必要はない。

### C-3. 位置・方位の正確性（現地検証）
